## Gap 1 — BLOCKER: Strain detail crash on null strain
File: `src/app/strains/[slug]/StrainDetailPageClient.tsx`
Root cause: `useQuery` succeeds but `detailData.strain` is null (invalid slug). Component reaches main JSX (line 520+) which accesses `strain.image_url` without null check. An early return for `!detailData?.strain` exists at lines 496-518, but an additional explicit `!strain` guard before the main JSX provides defense-in-depth.

## Gap 2 — MAJOR: FollowButton unfollow doesn't invalidate follow-status query
File: `src/components/social/follow-button.tsx`
Root cause: After unfollow success (lines 129-136), the code invalidates `followingKeys.list(user.id)` and `followersKeys.list(userId)` but NOT `['follow-status', userId]`. When `optimisticStatus` clears, `computedStatus` falls back to stale cached `fetchedStatus.is_following: true`, reverting the UI to "Following".

In `src/components/social/follow-button.tsx`, in the unfollow success block (around line 129-136), add `queryClient.invalidateQueries({ queryKey: ['follow-status', userId] })` after the existing invalidations.

The unfollow block (lines 129-136) currently reads:
```
if (!error) {
    setOptimisticStatus(null);
    onFollowChange?.();
    router.refresh();
    // Invalidate React Query cache
    queryClient.invalidateQueries({ queryKey: followingKeys.list(user.id) });

}
```

Change it to:
```
if (!error) {
    setOptimisticStatus(null);
    onFollowChange?.();
    router.refresh();
    // Invalidate React Query cache
    queryClient.invalidateQueries({ queryKey: followingKeys.list(user.id) });
    queryClient.invalidateQueries({ queryKey: followersKeys.list(userId) });
    queryClient.invalidateQueries({ queryKey: ['follow-status', userId] });
}
```

This ensures the follow-status query cache is cleared, so when `optimisticStatus` is cleared, `computedStatus` re-fetches fresh data instead of using stale `is_following: true`.
